# API Contract - simple-be-ocr

Last updated: 2026-03-24

## 1) Scope

This document defines request and response contracts for endpoint services implemented in this backend:
- GET /health
- POST /api/v1/ocr/process

## 2) Base URL

Environment-specific. Examples:
- Local: http://localhost:4000
- Staging/Production: use deployed host

## 3) Common Rules

### Authentication

- OCR endpoint requires Bearer token in Authorization header.
- Format: Authorization: Bearer <token>

Current implementation validates header presence and token shape only (starts with Bearer and non-empty).

### Content Types

- OCR endpoint accepts multipart/form-data.
- Health endpoint returns application/json.

### Error Envelope

All handled failures return:

```json
{
  "error": {
    "code": "STRING_CODE",
    "message": "Human readable message"
  }
}
```

## 4) GET /health

### Purpose

Service liveness probe.

### Request

- Method: GET
- Path: /health
- Auth: none
- Body: none

### Success Response

Status: 200

```json
{
  "status": "ok"
}
```

## 5) POST /api/v1/ocr/process

### Purpose

Parse uploaded document/image content into normalized expense transactions.

### Request

- Method: POST
- Path: /api/v1/ocr/process
- Auth: required Bearer token
- Content-Type: multipart/form-data
- Query Parameters:
  - sourceType (optional, string): Explicitly specify document type
    - receipt: force expense parser (Receipt/Invoice processing)
    - bank-notification: force document reader (Bank notification processing)
    - omitted: auto-detect based on content (default behavior)

Form fields:
- file (required, binary): image or document payload for OCR
- text (optional, string): currently ignored by backend parser, but accepted in multipart payload for frontend compatibility

Notes:
- If file is missing, request fails with INVALID_FILE.
- Upload size limit follows MAX_FILE_SIZE_BYTES (default 10 MB).
- When sourceType is specified, backend routes directly to the specified endpoint without auto-detection fallback.

### Success Response (200)

```json
{
  "requestId": "string",
  "provider": "mock | google | google-document-ai | other configured provider",
  "sourceType": "receipt | bank-notification",
  "rawText": "string",
  "parsed": {
    "merchant": "string",
    "transactionDate": "YYYY-MM-DD or provider string",
    "totalAmount": 0,
    "currency": "string",
    "category": "string",
    "paymentMethod": "string or null",
    "notes": "string or null",
    "lineItems": []
  },
  "transactions": [
    {
      "merchant": "string",
      "transactionDate": "YYYY-MM-DD or provider string",
      "totalAmount": 0,
      "currency": "string",
      "category": "string",
      "paymentMethod": "string or null",
      "notes": "string or null",
      "lineItems": []
    }
  ],
  "confidence": {
    "overall": 0,
    "fields": {
      "merchant": 0,
      "transactionDate": 0,
      "totalAmount": 0
    }
  },
  "raw": {
    "providerSpecific": "unknown shape"
  }
}
```

### Field Semantics

- requestId: unique ID generated by backend.
- provider: OCR provider used by current runtime config.
- sourceType:
  - receipt: receipt/invoice style text extraction
  - bank-notification: bank message style extraction (can produce multiple transactions)
- parsed: primary transaction object (first transaction for bank notifications).
- transactions: one or more normalized transaction objects.
- confidence.overall: confidence score 0..1 from provider heuristics.
- confidence.fields: per-field confidence scores.
- raw: passthrough/debug payload, provider-specific and not stable for strict client parsing.

### Parsing Behavior

#### sourceType Parameter Effect

When sourceType query parameter is provided:

- **sourceType=receipt**: Backend skips auto-detection and directly uses `_RECEIPT_PROCESS_LINK` (expense parser)
  - Suitable for: receipts, invoices, expense receipts
  - Behavior: Single pass through expense parser endpoint
  
- **sourceType=bank-notification**: Backend skips auto-detection and directly uses `_DOCUMENT_PROCESS_LINK` (document reader) followed by bank notification parser
  - Suitable for: bank SMS/push notifications, transaction messages
  - Behavior: Single pass through document reader + bank parser
  
- **sourceType omitted (auto-detect)**: Backend uses the default multi-pass strategy described below

#### A) Auto-Detect Flow (No sourceType Specified)

Backend flow for receipt:
- First pass uses `_RECEIPT_PROCESS_LINK` (expense parser).
- If parsed receipt has missing/zero `totalAmount`, backend retries with `_DOCUMENT_PROCESS_LINK`.
- If fallback response text matches bank notification pattern, backend classifies as `bank-notification` and parses via bank parser.

Backend then maps OCR entities to normalized fields:
- merchant <- merchant_name / merchant / supplier / vendor / seller
- transactionDate <- transaction_date / receipt_date / invoice_date / date
- totalAmount <- total_amount / amount_due / net_amount / balance_due / total
- currency <- currency / currency_code (fallback IDR)
- paymentMethod <- payment_method / payment_type / method
- notes <- notes / description / memo
- lineItems <- entities containing line_item

Fallbacks used when missing:
- merchant: Unknown merchant
- transactionDate: current date (YYYY-MM-DD)
- totalAmount: 0
- category: Uncategorized
- currency: IDR

Google endpoint config:
- `_RECEIPT_PROCESS_LINK`: expense parser endpoint (required, first pass)
- `_DOCUMENT_PROCESS_LINK`: document processor endpoint (required, fallback and bank notification parser source)

#### B) Bank Notification Source

Source detection uses raw text patterns:
- transaksi kartu kredit/debit
- sebesar IDR

When both patterns are found, backend classifies sourceType as bank-notification and extracts zero or more transactions from the message body.

Current parser target format example:
- Kartu Kredit Danamon 9223 di MERCHANT sebesar IDR 249.400,00 pada 23/03/26 10:29:32 telah berhasil

Normalization:
- totalAmount: Indonesian numeric format converted to number (249.400,00 -> 249400)
- transactionDate: DD/MM/YY converted to YYYY-MM-DD (23/03/26 -> 2026-03-23)
- paymentMethod: Credit Card or Debit Card + bank + last 4 digits

### Error Responses

- 400 Bad Request

```json
{
  "error": {
    "code": "INVALID_FILE",
    "message": "A file upload is required"
  }
}
```

- 401 Unauthorized

```json
{
  "error": {
    "code": "UNAUTHORIZED",
    "message": "Missing bearer token"
  }
}
```

- 500 Internal Server Error

```json
{
  "error": {
    "code": "OCR_PROCESS_FAILED",
    "message": "Unable to process document"
  }
}
```

Note: 500 message may include provider error text (for example missing Google endpoint or upstream OCR failure).

## 6) Stitch Screen Alignment

Current Stitch Add Expense flow shows two AI entry paths:
- OCR photo upload
- Quick Add text parsing

Backend currently implements OCR file processing only through POST /api/v1/ocr/process.
Quick Add text-only parsing is not implemented as a dedicated endpoint yet.

Recommended next endpoint for full Stitch parity:
- POST /api/v1/ocr/parse-text
- Input: text/plain or JSON with free-form transaction text
- Output: same normalized shape as transactions in /api/v1/ocr/process
